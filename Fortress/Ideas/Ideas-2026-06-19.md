# Ideas-2026-06-19: Documentation as Stateless Workflow Coordinator

**Date:** 2026-06-19  
**Status:** Research Synthesis  
**Purpose:** Summarize key papers, videos, patterns, and utilities supporting the **Stateless Workflow Coordinator Model** for AI coding agents.

---

## 1. Core Model (Recap)

The **Stateless Workflow Coordinator Model** proposes that high-quality documentation can function as a lightweight, stateless orchestrator/state-machine that AI agents must follow.

**Key Principles:**
- **Strict Separation**: Task/Spec = Data (“what” to build) vs. Workflow = Process (“how the agent must behave”).
- **Stateless Workflows**: The workflow definition itself carries no accumulated state. State lives in artifacts and context.
- **Strong Enforcement**: “Entice first, Spanish Inquisition when needed” — clear gates, named procedures, and violation paths.
- **Coordinator Mindset**: Top-level docs (e.g. `BuildWorkflow.md`) act as the executable control plane.
- Supporting structure: `.docs/Workflows/`, `Procedures/`, `TaskTemplates/`.

This model evolves beyond traditional spec-driven development by making **documentation itself the process layer**.

## 2. Key Papers

### 2.1 Spec Kit Agents: Context-Grounded Agentic Workflows
- **Link**: https://arxiv.org/abs/2604.05278
- **Summary**: Introduces a multi-agent system with a **state-machine orchestrator** managing stages (Specify → Plan → Tasks → Implement). Adds explicit discovery (grounding) and validation hooks. Uses structured artifacts (`SPEC.md`, `PLAN.md`, `TASKS.md`).
- **Relevance**: Closest existing implementation of a coordinator + structured artifacts with phase gates. Strong support for separation of generation vs. evaluation.
- **Key Strength**: Validation at stage boundaries prevents error propagation.
- **Limitation**: Still somewhat blends process into the SDD pipeline.

### 2.2 Talk Freely, Execute Strictly: Schema-Gated Agentic AI
- **Link**: https://arxiv.org/abs/2603.06394
- **Summary**: Proposes **schema-gated orchestration** where schemas act as mandatory execution boundaries. Strictly separates conversational planning from validated execution.
- **Relevance**: Excellent model for strong enforcement via gates/schemas. Aligns with “violation paths” and mandatory checks.
- **Key Strength**: Decentralized, stateless enforcement through schemas.
- **Limitation**: More schema/JSON-oriented than human-readable Markdown docs.

### 2.3 Open Agent Specification (Agent Spec)
- **Link**: https://arxiv.org/abs/2510.04173
- **Summary**: Declarative, framework-agnostic language for defining agents and **workflows as portable graphs** (Flows). Supports validation, composition, and adapters.
- **Relevance**: Provides a formal way to define reusable workflow components that could be expressed in documentation.
- **Key Strength**: Portability and built-in validation.
- **Limitation**: More infrastructure-focused than documentation-first.

### 2.4 A Practical Guide for Designing Production-Grade Agentic AI Workflows
- **Link**: https://arxiv.org/abs/2512.08769
- **Summary**: Engineering guide emphasizing separation of workflow logic from orchestration layers, externalized prompts, single-responsibility agents, and maintainability (KISS).
- **Relevance**: Supports clean separation and treating documentation-like artifacts as governable components.
- **Key Strength**: Practical maintainability advice.

## 3. Key Videos (Claude Code Focused)

### 3.1 How to Build $10,000 Agentic Workflows – Nate Herk
- **Link**: https://www.youtube.com/watch?v=vFepZE_wrfg
- **Key Insight**: Treats dedicated **Workflow Markdown files** as directly executable “recipes.” `CLAUDE.md` acts as persistent coordinator. Includes human review gates and self-healing.
- **Relevance**: Strongest real-world example of Markdown as executable workflow.

### 3.2 How to Properly Use Claude Code Agent Teams – Cole Medin
- **Link**: https://www.youtube.com/watch?v=uvs1Igr4u6g
- **Key Insight**: Lead agent + shared plan/task list for coordination. Contract-first approach before parallel work. Plans serve as central orchestrating artifacts.
- **Relevance**: Good model for multi-agent coordination with a plan document as coordinator.

### 3.3 How to Automate Your Life & Work w/ Claude Code – Liam Ottley
- **Link**: https://www.youtube.com/watch?v=2bsfQThGXxc
- **Key Insight**: Workspace templates + `/prime` (loads `CLAUDE.md` + context) + `/create plan` + `/implement` loop. Strong emphasis on reusable Markdown commands and fresh stateless sessions.
- **Relevance**: Closest match to priming + reusable documentation as coordinator.

### 3.4 Spec Driven Workflow with Claude Code (Plan Mode) – Net Ninja
- **Link**: https://www.youtube.com/watch?v=CmrmmszC_F4
- **Key Insight**: Plan Mode generates structured Markdown plans with steps, decisions, and verification criteria before implementation. Includes explicit review gate.
- **Relevance**: Clean tactical pattern for mandatory high-quality plan artifacts.

## 5. Synthesis & Recommendations

### Strongest Alignments
- **Spec Kit Agents** + **Liam Ottley video** → Best combination of state-machine orchestration + practical Markdown coordinator patterns.
- **Schema-Gated** paper → Strongest conceptual model for hard enforcement.
- **Nate Herk** + **Net Ninja** videos → Best examples of treating dedicated workflow/plan Markdown files as executable.

### Recommended Next Steps for `.docs/` Structure
1. Create `Workflows/BuildWorkflow.md` with explicit stages + gates (inspired by Spec Kit + Liam Ottley priming).
2. Build `Procedures/` folder with reusable steps (Context Grounding, Validation Gate, Violation Handling).
3. Use Plan Mode / structured plan generation as a mandatory early gate.
4. Consider adding lightweight schema-style validation for critical rules.
5. Keep `TaskTemplates/` clean as pure data (FeatureSpec, TaskSpec).

### Open Questions
- How much structure is ideal before it becomes brittle?
- Should enforcement be mostly soft (clear docs) or include automated validation hooks?
- Best balance between human-readable Markdown and machine-validatable formats?
