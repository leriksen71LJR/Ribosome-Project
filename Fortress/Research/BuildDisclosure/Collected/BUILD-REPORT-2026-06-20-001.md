## Key Implementation Decisions

### Architecture
- Full `Components/` tree with `Contracts/`, `Implementations/`, `Model/` per component.
- `IDependencyModule` in `Bootstrapping/` — only interface; all registrations in `Modules/`.
- `Program.cs` discovers and invokes modules only — no direct `services.Add*` in entry point.

### Handlers
- All 11 handlers use `Task<bool> ExecuteAsync(ActionContext, CancellationToken)` with guard clauses as first statements.
- `IsVisible()` rules follow `HANDLER_INVENTORY.md` exactly.
- Handler `Name` properties have no menu numbers — assigned at display time.
- `ArchiveCompletedHandler`: "archive" implemented as removal from active list (delete from storage), since `TaskItem` has no `IsArchived` field in the spec. Confirmed per storage design note in `HANDLER_INVENTORY.md`.

### Security
- `Batteries_V2.Init()` is the first statement in `Program.cs`.
- SQLCipher key applied via `PRAGMA key = "x'<hexkey>'"` (lowercase hex) per spec.
- Argon2id: `MemorySizeKb=65536`, `Iterations=4`, `Parallelism=1`, `Salt=16 bytes`.
- Salt and KDF params in `fortress.security.json`; key never stored.
- Wrong password: `VerifyKey` (open + PRAGMA + `sqlite_master` probe) returns false → `UnauthorizedAccessException` thrown. No new DB created.

### Storage
- Per-type tables (Tasks, Notes, Credentials, Goals) per `CODING_DESIGN.md` schema.
- `SqliteStorageService.EnsureSchemaAsync()` called by `SessionManager` after unlock.
- `EncryptionService` holds derived key in memory; `ApplyKeyAsync` applies it per connection (see Deviations #1).

### DI
- `SqliteStorageService` registered as concrete singleton in `InfrastructureModule`; mapped to `IStorageService` on same instance.
- `SecurityModule` constructs `SessionManager` manually to inject concrete `SqliteStorageService`.

## Deep Documentation Audit

### Mental Model Formed

Fortress is a single-user encrypted console vault. Components are organized by feature area under `src/Fortress.Console/Components/`, each with `Contracts/`, `Implementations/`, `Model/`. The session lifecycle (unlock → act → lock/timeout) is driven by `SessionManager`. Handlers receive `ActionContext` (current items + session state + errors) and operate on it. DI wiring flows exclusively through `IDependencyModule` implementations; `Program.cs` only discovers and invokes modules.

### Strengths

- **`HANDLER_INVENTORY.md`:** Every handler row translated directly to `Name`, `Description`, `IsVisible()` body, and constructor injection list. Zero interpretation required — this is the strongest executable pseudocode in the set.
- **`ARCHITECTURE_SECURITY.md` §Implementation Specification:** The hex PRAGMA format, `Batteries_V2.Init()` placement, Argon2id parameter table, `fortress.security.json` schema, wrong-password behavior table, and first-time vs. subsequent unlock flows were all implementation-ready. Produced correct code in one pass.
- **`CODING_DESIGN.md` §Database Schema:** The per-type SQL `CREATE TABLE` statements were directly usable. No schema invention required.
- **`AGENTS.md` Rule 6:** "Program.cs discovers and invokes modules only" produced the module array pattern exactly.

### Significant Issues

Per-connection key application (`ApplyKeyAsync`) is not specified. `SqliteStorageService` opens a new connection per operation and requires key application on each open. | High | Added `ApplyKeyAsync`, `IsInitialized`, `Clear()` to `IEncryptionService`. Reported as deviation. Different agents will produce different extensions. |

### Assumptions Made (Rule 11)

| # | Assumption | Why necessary | Severity | Validated? |
|---|------------|---------------|----------|------------|
| 1 | `IEncryptionService` needs `ApplyKeyAsync` for per-connection keying | Storage opens a new connection per operation; spec only covers database initialization | High | Functionally validated — build passes |
| 2 | `EncryptionService` holds derived key in memory after `InitializeDatabaseAsync` | No other mechanism exists to pass the key to subsequent connection opens | High | Consistent with singleton lifetime; key cleared on `LockAsync` |
| 3 | `ArchiveCompletedHandler` removes completed tasks from storage entirely | `TaskItem` has no `IsArchived` field; spec says "archive/delete per storage design" | Low | Consistent with spec wording and MVP scope |
| 4 | `fortress.security.json` absence + `fortress.db` absence = first-time setup | Spec defines this case explicitly | None — spec covers it |

### Recommendations

1. **`CODING_DESIGN.md` — complete `IEncryptionService` body:** Add `ApplyKeyAsync(SqliteConnection, CancellationToken)`, `IsInitialized`, `Clear()`. This is the single highest-impact gap — every agent will extend the interface differently.
2. **`CODING_DESIGN.md` — clarify `ArchiveCompletedHandler` storage behavior:** State whether archive = soft-delete (new `IsArchived` column) or hard-delete (remove row). The current wording leaves this open.
3. **`CODING_DESIGN.md` — add `ReadPassword()` to `IInputService` or acknowledge its absence:** If masking is desired, the interface must declare it. Currently, password entry echoes to console.
4. **`ARCHITECTURE.md` — update target framework or provide `global.json`:** Specify `.NET 10` to eliminate the framework deviation across agents.

### Overall Assessment

**Rating:** 8/10

**One-sentence justification:** The handler inventory and security spec were production-quality pseudocode that generated correct code directly; the only significant gap was `IEncryptionService` body incompleteness, which forced a Medium-risk interface extension.

**Would another agent produce the same structure from these docs alone?** Partially — folder structure, handler names, `IsVisible()` rules, security primitives, and DB schema would converge; `IEncryptionService` extensions would diverge.

## Open Gaps and Assumptions

1. `IEncryptionService.ApplyKeyAsync` — invented; other agents will likely differ.
2. `ArchiveCompletedHandler` archive = hard delete — no canonical definition in docs.
3. No password masking — `IInputService.ReadLine()` only; credentials entered in plaintext at console.
4. Integration tests not written — SQLCipher round-trip (write → lock → unlock → read) is untested at test suite level.
5. Auto-lock fires at loop iteration boundary — a long-running handler could delay lock by its execution time.
